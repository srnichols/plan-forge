Here's a conversation I've had at least a dozen times:

"Does Plan Forge work with Claude?" "What about Cursor?" "We use Gemini CLI — can we still use your guardrails?"

The answer used to be "sort of — copy the instruction files and adapt them." That was slow, fragile, and nobody did it consistently. So we built **native adapters for every major AI coding agent.** One setup command. Seven agent formats. Same guardrails everywhere.

THE PROBLEM: TEAMS DON'T AGREE ON TOOLS

Your team doesn't all use the same AI coding tool. One dev swears by Copilot in VS Code. Another prefers Cursor's inline editing. The senior architect uses Claude Code for deep reasoning. The DevOps engineer likes Gemini CLI. The intern just discovered Windsurf.

This is normal. Different tools have different strengths. **The problem isn't tool diversity — it's guardrail fragmentation.**

If your guardrails only work in one tool, most of your team isn't covered. And the team members who aren't covered are the ones who produce the code that fails review. The PRs that don't match your patterns. The architecture that drifts in a different direction.

You can't enforce "everyone uses the same tool" on a team of experienced developers. But you can enforce **"everyone gets the same guardrails" regardless of tool.**

ONE COMMAND, SEVEN AGENTS

One setup command generates native files for every supported agent. Not generic text files — **actual native configuration that each tool reads automatically at session start.**

[IMAGE: 06-one-to-seven.png — Diagram: Single "pforge init" command at top, branching to 7 agent format outputs: Copilot (.github/), Claude (CLAUDE.md), Cursor (.cursor/), Windsurf (.windsurf/), Codex (.agents/), Gemini (GEMINI.md), Generic (.ai/)]

Here's what each agent gets:

**Copilot (default):**
Auto-loading .github/instructions/, agents, prompts, skills, MCP tools. This is the gold standard — edit a database file and database guardrails load automatically through applyTo frontmatter.

**Claude Code:**
CLAUDE.md with all guardrails embedded inline + .claude/skills/ for every prompt and agent + MCP configuration. Claude reads CLAUDE.md at session start. Every guardrail is there from the first token.

**Cursor:**
.cursor/rules with all guardrails + .cursor/commands/ for pipeline steps and reviewers. Type planforge.step3-execute-slice and Cursor runs it natively.

**Windsurf:**
.windsurf/rules/ + .windsurf/workflows/ for all pipeline and review commands. Windsurf's workflow format maps naturally to Plan Forge's pipeline steps.

**Codex CLI:**
.agents/skills/ with SKILL.md for every prompt and agent. Codex discovers and runs skills automatically.

**Gemini CLI:**
GEMINI.md with @import syntax + .gemini/commands/ as TOML + MCP via settings.json. Gemini's import mechanism pulls in guardrails modularly.

**Generic:**
.ai/AI_INSTRUCTIONS.md — works with ChatGPT, Perplexity, Ollama, anything that accepts a system prompt. Copy-paste it into any chat window.

**Every format includes all 18 guardrail files, all pipeline prompts, and all 19 reviewer agents** — transformed into the native format each tool expects.

HOW THE ADAPTERS WORK

Each adapter takes Plan Forge's canonical files and transforms them into whatever format the AI tool reads natively. The canonical files are the source of truth. Adapters are generated, not hand-maintained.

[IMAGE: 06-adapter-flow.png — Flow diagram: Canonical guardrail files (center) → Transform → 7 native format outputs, each with their specific file paths and formats]

This means when you update a guardrail — say you add a new security rule — you run the setup command again and **every agent format gets the update automatically.** No manual syncing. No "I forgot to update the Cursor rules." One source of truth, seven outputs.

THE KILLER FEATURE: UNIFIED MEMORY ACROSS AGENTS

Here's the part most people miss — and it might be the biggest deal of all.

When you wire in OpenBrain (https://github.com/srnichols/OpenBrain) as a shared MCP server, **agents in completely different tools share the same memory.** Think about what that means:

**1.** Your Claude Code session hardens the plan and captures architectural decisions to OpenBrain

**2.** Your Cursor session executes the slices — and searches OpenBrain first, **finding every decision Claude made**

**3.** Your Copilot session runs the independent review — and queries OpenBrain for the full decision history from both previous sessions

Three different tools. Three different AI models. **One shared institutional memory.**

[IMAGE: 06-unified-memory.png — Sequence diagram: Claude Code (Session 1: Plan) → writes to OpenBrain → Cursor (Session 2: Build) reads from OpenBrain, writes new decisions → Copilot (Session 3: Review) reads full history from OpenBrain]

This isn't just "same guardrails everywhere." This is **cross-agent collaboration through persistent semantic memory.** The Claude session doesn't know Cursor exists. The Cursor session doesn't know Claude was involved. But they're both working from the same decision history because OpenBrain sits underneath all of them.

It doesn't matter which agent writes the code. What matters is that **every agent knows what every other agent decided — and why.**

This is what makes the multi-agent story more than a convenience feature. Your whole team's AI agents build on each other's knowledge, across tools, across sessions, across time. The decisions compound. The mistakes are learned from. And **nobody starts from zero — ever.**

WHY THIS MATTERS FOR TEAMS

When every developer's AI agent loads the same architecture principles, the same security rules, and the same testing standards — regardless of tool — **the code converges on the same quality bar.**

No more "this PR doesn't match our patterns because I used a different tool." The guidelines load automatically in every tool. The agent can't avoid them.

Here's what changes in practice:

• **New hire on Cursor?** — They have every guardrail from day one. No onboarding lag.
• **Senior dev on Claude?** — All 19 reviewer agents are available. Same depth of analysis.
• **Experimenting with Gemini?** — MCP auto-configures. Drop in and go.
• **Using everything?** — One command and every format generates in one pass.
• **Switching tools?** — Your guardrails, memory, and pipeline work identically in the new tool.

The organizational benefit is significant. Code reviews become faster because **every PR already follows the same patterns,** regardless of which tool produced it. New hires are productive faster because the guardrails teach them the codebase conventions in real-time. Architecture stays consistent because the constraints are structural, not cultural.

THE TECHNICAL DETAILS

For the engineers who want specifics, here's what the adapter generation does:

**File count per agent:**
• Copilot: 40+ files (.github/instructions/, agents/, prompts/, skills/)
• Claude: 1 composite + ~25 skill files
• Cursor: ~20 rule files + ~15 command files
• Windsurf: ~20 rule files + ~10 workflow files
• Codex: ~25 skill files
• Gemini: 1 composite + ~15 TOML command files
• Generic: 1 comprehensive markdown file

**Generation time:** Under 5 seconds for all formats.

**Update workflow:** Edit canonical files → run setup → all formats regenerate. Zero manual work.
