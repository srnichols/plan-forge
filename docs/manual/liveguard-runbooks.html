Severity Matrix

Every LiveGuard alert carries one of four severity levels. The matrix below defines response SLA and escalation path. Full runbooks per alert type follow.

Severity Response SLA Notify Dashboard Badge
Critical Immediate — within 1 hour On-call + team lead Red badge on Triage tab
High Same business day On-call engineer Amber badge on Triage tab
Medium Next sprint Team chat Yellow dot on relevant tab
Low Backlog — No badge

Runbook Index

Full runbooks shipping in v2.28.0 — one per major alert type:

  • Drift spike — architecture diverged from plan baseline
  • Secret found — high-entropy string detected in a committed diff
  • Env diff gap — required key missing from one or more environment files
  • Regression gate failure — previously passing gate now fails
  • Dependency vulnerability — new CVE in a watched package
  • Incident MTTR exceeded — time-to-resolve beyond threshold

Fix Proposal Workflow (v2.29)

When a LiveGuard tool fires a failure (regression, drift, incident, or secret found), forge_fix_proposal generates a scoped 1-2 slice fix plan for human review. This is the detect → propose → approve → fix loop.

  1. Trigger: Run pforge fix-proposal --source regression (or drift/incident/secret) after the alert fires.
  2. Review the plan: Open docs/plans/auto/LIVEGUARD-FIX-<incidentId>.md. The plan contains the failing command, affected files, and a template fix slice with <!-- TODO --> markers for you to fill in.
  3. Fill in the fix: Complete the TODO markers in the fix slice. For secret findings, the template directs you to remove the credential from the diff and rotate it externally before proceeding.
  4. Execute on a branch: pforge run-plan --assisted docs/plans/auto/LIVEGUARD-FIX-<incidentId>.md. The plan targets a dedicated branch — never master.
  5. Verify: The second slice re-runs the exact commands that originally failed. Green gate = fix confirmed.
  6. Promote or close: Merge the branch if the fix holds. Close the proposal by updating its status in .forge/fix-proposals.json. Auto-generated plans in docs/plans/auto/ are gitignored — promote manually to docs/plans/ if you want to keep it in version history.
Loop cap: forge_fix_proposal generates at most one proposal per incidentId. If the first proposal doesn’t resolve the issue, address it manually — the tool will return status: "needs-human-intervention" on the second call.
