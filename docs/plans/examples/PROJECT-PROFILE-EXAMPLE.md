## Data Access Rules

- **ALL queries MUST use Dapper with parameterized SQL** ŌĆö no EF Core, no string interpolation
- **ALL queries MUST include `WHERE tenant_id = @TenantId`** except for global reference tables (users, system config)
- **SELECT only needed columns** ŌĆö never `SELECT *`
- **Use column aliases** for clarity: `SELECT u.display_name AS DisplayName`
- **Batch operations** when inserting/updating multiple records ŌĆö never loop individual queries
- **Repository methods return domain objects** ŌĆö never raw `dynamic` or `DataTable`
- **Connection management**: Use `IDbConnectionFactory` ŌĆö never create `NpgsqlConnection` directly

### Table Pattern
```sql
-- Every tenant table has:
CREATE TABLE entity_name (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID NOT NULL REFERENCES tenants(id),
    -- business columns
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_by UUID REFERENCES users(id),
    updated_by UUID REFERENCES users(id)
);

-- Always add RLS policy:
ALTER TABLE entity_name ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON entity_name
    USING (tenant_id = current_setting('app.tenant_id')::uuid);

-- Always add tenant index:
CREATE INDEX idx_entity_name_tenant ON entity_name(tenant_id);
```

## API Conventions

- **URL versioning**: `/api/v1/resource`
- **Response wrapper**: All responses use `ApiResponse<T>` (success) or `ProblemDetails` (error)
- **Pagination**: `pageNumber` and `pageSize` query params, return `PagedResult<T>` with total count
- **Default page size**: 20, max: 100 (enforced server-side)
- **Date format**: ISO 8601 (`2026-03-24T14:30:00Z`) ŌĆö never locale-specific
- **IDs**: UUIDs as strings ŌĆö never expose sequential integers
- **Error responses**: RFC 7807 ProblemDetails with `type`, `title`, `status`, `detail`
- **Authorization**: `[Authorize]` on all endpoints, `[AllowAnonymous]` only on public endpoints (login, register, health)

## Async Patterns

- **ALL I/O operations MUST be async** (`async Task<T>` return type)
- **ALL async methods MUST accept `CancellationToken ct = default`** as last parameter
- **NEVER block on async**: No `.Result`, `.Wait()`, `.GetAwaiter().GetResult()`
- **Use `await using`** for disposable async resources (DB connections, HTTP clients)
- **ConfigureAwait**: Not needed in ASP.NET Core (no SynchronizationContext)

## Caching Conventions

- **Cache keys MUST include tenant ID**: `tenant:{tenantId}:entity:{entityId}`
- **Use configured TTL profiles** ŌĆö never hardcode TTL values:
  - `StaticData`: 24 hours (production)
  - `TenantSettings`: 1 hour
  - `UserSession`: 30 minutes
  - `ProducerListings`: 15 minutes
- **Cache-aside pattern**: Read from cache ŌåÆ miss ŌåÆ read from DB ŌåÆ write to cache
- **Invalidate on write**: Always clear relevant cache entries after create/update/delete

## Error Handling

- **Global exception handler** (`IExceptionHandler`) maps exceptions to HTTP status codes
- **Custom exception types** for domain-specific errors:
  - `TenantNotFoundException` ŌåÆ 404
  - `TenantAccessDeniedException` ŌåÆ 403
  - `BusinessRuleViolationException` ŌåÆ 422
  - `ConcurrencyConflictException` ŌåÆ 409
- **Never throw generic `Exception`** ŌĆö use typed exceptions
- **Never catch generic `Exception`** unless re-throwing ŌĆö catch specific types
- **Always include context** in exception messages: tenant ID, entity ID, operation name

## Testing Rules

- **Unit tests**: xUnit, one test class per service, Arrange-Act-Assert pattern
- **Integration tests**: MSTest, shared PostgreSQL container via `SharedContainerTestBase`
- **E2E tests**: NUnit + Playwright, categorized (Desktop, Mobile, Navigation, Stress)
- **Test naming**: `MethodName_Scenario_ExpectedResult` (e.g., `GetByIdAsync_ValidId_ReturnsEntity`)
- **Mock external dependencies only** ŌĆö never mock internal services (test those with integration tests)
- **Write test FIRST** (TDD) for all service-layer methods

## Performance Rules

- **Use `FrozenDictionary` / `FrozenSet`** for read-heavy lookup data loaded at startup
- **Use source-generated logging** (`[LoggerMessage]`) for hot paths ŌĆö zero allocations
- **Use `Span<T>`** for string manipulation in hot paths
- **Server-side filtering** for datasets >100 records ŌĆö never filter client-side on large sets
- **Only optimize after profiling** ŌĆö never guess at bottlenecks

## Multi-Tenant Security

- **4-layer tenant isolation** (enforced at each layer independently):
  1. **Database**: RLS policies on every tenant table
  2. **Repository**: `WHERE tenant_id = @TenantId` in every query
  3. **Service**: `GetCurrentTenantIdAsync()` validates tenant access
  4. **API**: `TenantValidationMiddleware` extracts tenant from JWT
- **Global reference tables** (no `tenant_id`): `users`, `system_config`, `subscription_plans`
- **Cross-tenant queries**: Only via GraphQL admin endpoints with `saas_admin` role
- **Cache isolation**: All cache keys prefixed with tenant ID

## Logging Conventions

- **Structured logging** with named properties: `logger.LogInformation("Created {Entity} {EntityId} for tenant {TenantId}", ...)`
- **Source-generated** (`[LoggerMessage]`) for performance-critical paths
- **Correlation IDs** on every log entry (from `Activity.Current?.Id`)
- **No PII in logs** ŌĆö mask email addresses, never log passwords or tokens
- **Log levels**:
  - `Debug`: Method entry/exit, detailed data
  - `Information`: Business events (user registered, order placed)
  - `Warning`: Recoverable issues (cache miss, retry attempt)
  - `Error`: Unrecoverable failures (DB down, auth failure)
  - `Critical`: System-level failures (startup failure, data corruption)
