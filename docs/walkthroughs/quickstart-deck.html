Your First Feature
with Plan Forge

Run the full pipeline in ~30 minutes.
Build a GET /health endpoint — tiny feature, complete pipeline.

~30 min VS Code + Copilot Full pipeline

What You'll Learn

By the end, you'll have run every step of the pipeline.

  • Specified a feature with Step 0
  • Created and hardened a plan (Steps 1–2)
  • Executed the plan slice-by-slice (Steps 3–4)
  • Run an independent review (Step 5)
  • Shipped the feature (Step 6)

This walkthrough uses the Prompt Template approach — attach .prompt.md files in Copilot Chat. It's the most educational path. After finishing, try the Pipeline Agents (handoff buttons) for a smoother experience.

Before You Start

Install the VS Code plugin, then validate your setup.

  1. 1 Install the VS Code extension (v1.113+): Install in VS Code  ·  Insiders build
  2. 2 Validate setup: 
.\validate-setup.ps1
# or
.\pforge.ps1 smith    # environment + health
.\pforge.ps1 check    # guardrail files

All checks should pass. If not, re-run .\setup.ps1 -Preset <your-stack>.
Using Claude, Cursor, or Codex? Add -Agent claude (or cursor, codex) to generate native files.

Step 0

Specify the Feature

Attach the prompt, answer the interview, get a hardened spec.

  1. 1Open Copilot Chat: Ctrl+Shift+I (Win) / Cmd+Shift+I (Mac)
  2. 2Select Agent Mode
  3. 3📎 Attach .github/prompts/step0-specify-feature.prompt.md
  4. 4Replace <FEATURE-NAME> with health-endpoint and send
  5. 5When asked "existing doc?" → say no. Answer the interview prompts.

The agent compiles your answers into a spec and creates docs/plans/Phase-1-HEALTH-ENDPOINT-PLAN.md.
When it says "Specification complete" — move to Session 1 Steps 1–2.

Steps 1–2

Pre-flight & Harden

Same session. Verify prerequisites, then lock the plan.

  1. 1 📎 Attach step1-preflight-check.prompt.md → replace plan name → send.
    Checks git state, guardrail files, and roadmap.
  2. 2 📎 Attach step2-harden-plan.prompt.md → replace plan name → send.
    Adds Scope Contract, Execution Slices, Validation Gates, Definition of Done.

The agent adds the 4 mandatory blocks to your plan.
When it says "Plan hardened" — Session 1 is complete. Start a new chat session.

Steps 3–4

Execute & Sweep

Three modes — pick the one that fits your workflow.

⚡ Automatic (recommended)

Orchestrator runs all slices, validates gates, tracks tokens, and reports results.

pforge run-plan docs/plans/Phase-1-HEALTH-ENDPOINT-PLAN.md

Monitor live at localhost:3100/dashboard

🤝 Assisted

You code each slice in VS Code; orchestrator validates gates automatically.

pforge run-plan --assisted docs/plans/Phase-1-HEALTH-ENDPOINT-PLAN.md

🖐 Manual

New chat session → Agent Mode → attach step3-execute-slice.prompt.md → replace plan name → send.

Agent executes Slice 1, then Slice 2, then runs a Completeness Sweep (TODO/FIXME scan).

Step 5

Independent Review

Fresh session. The reviewer never saw the builder's context.

  1. 1Start a new chat session — click +
  2. 2Select Agent Mode (or Ask Mode for read-only)
  3. 3📎 Attach step5-review-gate.prompt.md → replace plan name → send

The agent checks:

  • No forbidden files touched
  • Code follows architecture patterns
  • Tests included
  • No scope creep

For a health endpoint, this should be a clean PASS.

Step 6

Ship It

New session (or continue). Commit, update docs, capture lessons.

  1. 1Start a new chat session (or continue if context allows)
  2. 2📎 Attach step6-ship.prompt.md → replace plan name → send

The agent will:

  • Commit: feat(health): add GET /health endpoint
  • Update DEPLOYMENT-ROADMAP.md → Phase 1 ✅ Complete
  • Capture a brief postmortem
  • Ask whether to push — say yes or skip

🎉 Congratulations — you've run the full Plan Forge pipeline!

What Just Happened?

Four isolated sessions, each with a single responsibility.

Session 1 (Specify & Plan)
  → You described what you wanted
  → The AI structured it into a hardened plan

Session 2 (Execute)
  → The AI built it slice-by-slice
  → Validation gates at every boundary

Session 3 (Review)
  → Fresh AI session — no builder bias
  → Checked for mistakes and drift

Session 4 (Ship)
  → Committed, updated docs, captured lessons

Each session was isolated — the reviewer carried no bias from the builder, and every step loaded guardrails automatically from .github/instructions/.
